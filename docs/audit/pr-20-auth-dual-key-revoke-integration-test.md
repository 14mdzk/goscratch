## Test Scenario

1. Start Postgres + Redis testcontainers via `testutil.StartPostgres` / `testutil.StartRedis`.
2. Boot full app via `testutil.NewTestApp`.
3. Create a direct Redis client on the same address for key inspection.
4. Seed a user (`INSERT INTO users ...`) and `POST /auth/login` â†’ obtain `{access_token, refresh_token}`.
5. Assert `refresh:tok:<sha256(refresh)>` exists with `TTL > 0`.
6. Assert `refresh:user:<userID>:<sha256(refresh)>` exists with `TTL > 0`.
7. Assert both TTLs match within 2 s (set with the same `RefreshTokenTTL`).
8. `POST /users/me/password` (with Authorization header) â†’ ChangePassword triggers `RevokeAllForUser`.
9. Assert lookup key `EXISTS == 0`.
10. Assert index key `EXISTS == 0`.
11. `POST /auth/refresh` with old token â†’ assert HTTP 401.

## Design Notes

- Key shapes are re-derived in the test from first principles (same `sha256` + `hex.EncodeToString` logic), deliberately duplicated rather than imported, so a future rename in production code breaks the test rather than silently allowing it to pass.
- The access token used for the ChangePassword endpoint is synthesized with the correct `Issuer: "goscratch"` and `Audience: "goscratch-api"` required by `DefaultAuthConfig`. The login response token has empty issuer/audience because `testutil.TestJWTConfig` does not set them â€” a pre-existing gap noted below.
- `doRequest` / `parseBody` helpers are local to this test file.

## Pre-existing Infrastructure Issue (surfaced, not fixed)

`internal/platform/testutil/containers.go` starts `"postgres:17-alpine"`. The embedded migration `000001_init_users.up.sql` uses `uuidv7()` which was introduced in **PostgreSQL 18**, not 17. All integration tests in the repo fail for this reason. The fix (bumping to `"postgres:18-alpine"`) is outside this PR's scope (testutil infra refactor guard). Recommend adding a new punch-list row.

Additionally, `testutil.TestJWTConfig()` does not set `Issuer` or `Audience`. Tokens generated by `POST /auth/login` therefore have empty `iss`/`aud` claims, which the auth middleware (hardcoded `JWTIssuer: "goscratch"`, `JWTAudience: "goscratch-api"`) rejects. This silently breaks any integration test that uses the login-returned access token for auth-gated endpoints. This PR works around it by synthesizing the access token inline with correct claims.

Both findings are outside this PR's scope and should be addressed in a separate testutil-fix PR.
