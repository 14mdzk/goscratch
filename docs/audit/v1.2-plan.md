# v1.2 Plan — Production-Readiness Follow-ups

**Status:** plan-only. No code changes in this PR. Per-PR scope files (`pr-13-*.md` … `pr-22-*.md`) will be drafted as each PR is actively scoped, not up-front.

**Goal:** turn `goscratch` from "audited and hardened" (v1.1.0) into "operable in production by someone who is not its author." v1.1 closed correctness and security defects in the code; v1.2 closes the gaps surrounding the code (operator runbooks, supply-chain scanning, dependency-drift control, integration-test gaps left after the v1.1 hardening).

## Methodology

Three tiers, each independently shippable. The same overengineer guard from `2026-05-02-preship-audit.md` applies:

> Reject: abstractions without a 2nd consumer, config knobs nobody flips, generic future-proof interfaces, frameworks where stdlib works, defense-in-depth where one layer is already correct, hypothetical nation-state threats.

Tier A (security/CI) ships first because it has zero behavioural surface. Tier B (operator) is the production-readiness core. Tier C (test gaps) closes coverage debt left by the v1.1 hardening rush.

### PR-13 — Health readiness probe wired

`/health` currently returns `{"status":"ok"}` regardless of whether Postgres / Redis / RabbitMQ are reachable. Production load balancers cannot distinguish "process running" from "process able to serve". Split into:

- `GET /healthz/live` — `200` always (process up). Used by liveness probes.
- `GET /healthz/ready` — `200` when DB ping + cache ping (if enabled) + queue passive declare (if enabled) + casbin `Enforcer != nil` all pass within a 2s timeout. `503` otherwise. Used by readiness probes.
- `GET /health` — back-compat alias for `/healthz/live` (existing callers).

All sub-checks run in parallel with their own per-check budget. Report which sub-check failed in the response body for ops visibility (no leak: just check name + boolean).

### PR-14 — OpenAPI drift sync

Audit the spec against v1.1 behaviour:

- `/auth/logout` requires `Authorization: Bearer <access_token>` — add `security: [{ bearerAuth: [] }]`.
- `POST /api/files/upload` validates `Content-Type` against the allowlist via `http.DetectContentType` and rejects mismatches with HTTP 415 — add `415` response with `apperr.UnsupportedMediaType` schema.
- `POST /auth/login` and `POST /auth/refresh` are rate-limited (20 req / 5 min per IP); other routes use the global rate limiter — add `429 TooManyRequests` + `RateLimit-Limit`, `RateLimit-Remaining`, `RateLimit-Reset` headers on rate-limited operations. Define a reusable `responses.TooManyRequests`.
- `/auth/refresh` request body description: state explicitly that `user_id` is **not** accepted; the server resolves the userID from the lookup key. Extra fields are ignored, not validated.
- `/metrics` is **not** part of the OpenAPI surface (intentional — it now binds to `127.0.0.1:<port>` since v1.1.0). Confirm absent.

Regenerate `/docs` (Scalar) and verify in-browser.

### PR-17 — `docs/RUNBOOK.md`

One section per operational scenario. Each section: trigger condition, command sequence, verification step, rollback. Sections:

1. **Rotate `JWT_SECRET`** — flip the secret, restart, all current access tokens invalidated, all current refresh tokens invalidated (via auth.RevokeAllForUser sweep), users re-login. Communication template.
2. **Mass refresh-token revoke** — single user (`POST /api/users/me/password` already does this) and all users (run a maintenance job or `redis-cli` SCAN + DEL on `refresh:*`).
3. **Reload Casbin policies on demand** — bump the watcher channel (`PUBLISH casbin:policy:update <full-reload-msg>`); back-stop tick will reload within `ReloadInterval` regardless.
4. **Audit-log retention re-run** — dispatch the `audit.cleanup` job with a custom `retention_days` payload.
5. **Disable a compromised user fast** — `PATCH /api/users/<id>/deactivate` + `RevokeAllForUser`.
6. **Cache flush during incident** — when in doubt, flush the auth namespace, NOT the whole DB.
7. **Reading audit logs** — query patterns for "failed logins for user X", "all writes by user Y in window", "all `LOGIN` failures from IP Z".
8. **Reading metrics during incident** — top-10 latency endpoints, rate-limit rejections, queue depth, casbin reload count.

### PR-18 — Audit-log retention scheduler

Today: `audit_cleanup_handler` consumes `audit.cleanup` jobs but nothing **dispatches** them. Operators must wire external cron (`/jobs/dispatch` from a cron container). Two viable shapes:

- **A** — Document the external-cron pattern in the RUNBOOK and ship a sample `deploy/docker/cron-dispatch/` container. Zero code change. Smaller blast radius.
- **B** — Add an in-process scheduler tick to the worker (`scheduler.Tick(jobType, interval)`) that publishes `audit.cleanup` once per `audit.retention_interval` config (default 24h). Adds a code surface and a config knob.

Decision deferred to PR scope. **Default lean: A** (matches the "no abstraction without a 2nd consumer" guard — no other periodic jobs exist).

### PR-19 — Casbin watcher channel versioning

`internal/adapter/casbin/redis_watcher.go` hard-codes channel name `casbin:policy:update` (configurable, but the message format is implicit). When the message protocol evolves, two API instances on different versions during a rolling deploy will misparse each other's messages. Fix:

- Channel constant becomes `casbin:policy:update:v1`.
- New protocol versions bump the suffix; old + new instances stop cross-talking automatically.
- `Config.WatcherChannel` keeps the existing knob; the default reflects the version.

One-line code change + one test asserting the constant.

### PR-20 — Auth dual-key revoke integration test

Postgres + Redis testcontainer. Scenario:

1. Login → access + refresh.
2. Assert `refresh:tok:<sha256>` and `refresh:user:<userID>:<sha256>` both present with matching TTL.
3. Change password.
4. Assert both keys gone, refresh now returns 401.

### PR-21 — Casbin watcher e2e test

Two parallel `Adapter` instances sharing one Postgres + one Redis. Mutate policy on instance A, assert instance B sees the change within 1s. Repeat with `MemoryWatcher` (single-instance) and `RedisWatcher` (multi-instance). Asserts the incremental-apply path, not just `LoadPolicy`.

### PR-22 — Worker shutdown wg race test

Inject a slow handler (`time.Sleep(2s)` in `Process`). Trigger shutdown 100ms in. Assert the process exits in < `slow_handler_duration + budget`, not before. Asserts `wg` covers the active window. Repeat with the retry path mid-backoff: assert the retry timer cancels on `ctx.Done()` instead of waiting the full backoff.

## Out of Scope (still rejected)

Same v1.1 list, unchanged:

- Circuit breakers on cache / queue.
- Generic event bus to replace direct queue calls.
- Plugin system for adapters.
- gRPC alongside REST (see `docs/VISION.md`).
- Multi-tenancy.

Plus newly rejected for v1.2:

- **Distributed tracing across services** — there is one service. OTel is already wired for in-process spans.
- **OAuth provider integration** — out of scope per VISION; JWT is the auth model.
- **Webhook delivery system** — speculative; no consumer.

## Acceptance Per PR (v1.2)

Same five rules from the v1.1 punch-list, restated:

1. Tests added or updated for changed behavior. Plan-only PRs (this one) exempt.
2. `make lint test` clean.
3. `docs/features/<feature>.md` updated if behaviour changes.
4. `CHANGELOG.md` `[Unreleased]` entry.
5. Security-affecting PRs (#13, #14, #15, #19): include a "what an operator must change after upgrade" note in the PR body.

## Sequencing

Tier B is partially serial: PR-17 (RUNBOOK) references operations introduced or formalized by PR-18 and PR-19, so PR-17 ships **last** in Tier B.

Tier C is fully parallel.

Cross-tier ordering: Tier A → Tier B → Tier C. Each tier's release is a CHANGELOG cut (no separate version tag — single `v1.2.0` after all tiers land).
