## Design Decisions

- **EWKB format**: SRID-tagged (type flag `0x20000000`), little-endian, 25 bytes: 1 byte order + 4 type + 4 SRID + 8 lon (X) + 8 lat (Y).
- **pgx version**: v5 (`github.com/jackc/pgx/v5 v5.9.2`). No global codec registration needed — plain `sql.Scanner` + `driver.Valuer` suffice.
- **`Value()` returns `string`, not `[]byte`**: pgx v5 maps `[]byte` to OID 17 (bytea); PostGIS rejects bytea for geography columns with "parse error - invalid geometry". Returning a lowercase hex-encoded EWKB string causes pgx to use OID 25 (text), which PostGIS accepts as its geography text-input form. The production query shape `INSERT INTO t (loc) VALUES ($1)` with `$1 = geo.Point` works without any wrapper function.
- **`Scan` handles both protocols**: pgx text protocol delivers hex-encoded EWKB as `string`; binary protocol delivers raw EWKB as `[]byte`. Both are decoded by `DecodeEWKB`.
- **Lon = X, Lat = Y**: matches PostGIS / GeoJSON / WKT and C1's `Point{Lon, Lat}` field order.
- **NaN/Inf guard**: `EncodeEWKB` returns `ErrNaNOrInf` before writing any bytes.
- **SRID=0 rejection when SRID flag is set**: SRID=0 ("unknown") when the SRID flag is present is ambiguous for a `geography(Point,4326)` column. `DecodeEWKB` rejects it with `ErrInvalidWKB`. Plain WKB (no SRID flag) is accepted without restriction.
- **Integration test shape**: `INSERT INTO geo_test (loc) VALUES ($1)` / `SELECT loc FROM geo_test WHERE id = $1` — no ST_* wrapper functions. Spins its own `postgis/postgis:18-master` container independent of `testutil.StartPostgres`.
